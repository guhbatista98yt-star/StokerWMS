ESCOPO OBRIGATÓRIO DA AUDITORIA

1) SEGURANÇA
Analise profundamente:
- autenticação
- autorização
- segregação por perfil
- segregação por módulo
- segregação por empresa
- controle de acesso quebrado
- IDOR
- privilege escalation
- SQL Injection
- XSS
- CSRF
- SSRF
- path traversal
- command injection
- mass assignment
- enumeração de registros
- exposição de endpoints sensíveis
- vazamento de credenciais
- segredos hardcoded
- tokens inseguros
- sessão insegura
- cookies inseguros
- CORS incorreto
- headers de segurança ausentes
- rate limit ausente
- brute force
- logs expondo dados sensíveis
- respostas detalhadas demais para o cliente
- dependências vulneráveis
- bibliotecas desatualizadas e perigosas
- armazenamento inseguro no frontend
- manipulação insegura de localStorage/sessionStorage
- qualquer dado sensível exposto no client

2) CONCORRÊNCIA, CONSISTÊNCIA E INTEGRIDADE OPERACIONAL
Analise profundamente:
- race conditions
- competição entre operadores
- inconsistência entre múltiplas sessões
- múltiplas abas
- stale responses
- promises atrasadas contaminando tela atual
- concorrência sobre o mesmo pedido
- locking lógico e físico
- idempotência
- deduplicação de requisições
- double click
- double submit
- reentrada indevida
- estados presos
- estados órfãos
- estados “fantasma”
- contaminação de contexto entre pedidos
- reuso indevido de estado
- limpeza incompleta ao trocar de pedido
- avanço de status sem garantia transacional
- falha de rollback
- atualização parcial
- falta de transações em fluxos sensíveis
- inconsistência entre banco, backend e frontend
- bugs de fluxo que só aparecem em produção real

3) PERFORMANCE E ESCALABILIDADE
Audite:
- gargalos de API
- gargalos de banco
- N+1 queries
- falta de índices
- consultas mal estruturadas
- serialização excessiva
- overfetching
- processamento desnecessário
- renderizações excessivas
- componentes pesados
- cálculos repetidos sem necessidade
- polling mal implementado
- ausência de debounce/throttle quando necessário
- uso ruim de memória
- vazamentos de memória
- tempo de resposta nos módulos críticos
- comportamento sob carga
- pontos que podem degradar quando o uso crescer

4) ARQUITETURA E QUALIDADE DE CÓDIGO
Procure e corrija:
- funções grandes demais
- regras de negócio espalhadas
- duplicação de lógica
- acoplamento excessivo
- responsabilidades misturadas
- controller fazendo regra de negócio
- frontend decidindo regra crítica indevidamente
- ausência de camadas claras
- ausência de contratos fortes
- tipos frouxos
- código implícito e difícil de manter
- nomes ruins
- tratamento inconsistente de erros
- falhas de padronização
- pontos frágeis para manutenção futura
- código que parece funcionar mas é perigoso
- trechos que um sênior experiente reescreveria imediatamente

5) BANCO DE DADOS E TRANSAÇÕES
Verifique:
- constraints ausentes
- índices ausentes
- chaves frágeis
- inconsistência de status
- integridade referencial
- risco de escrita parcial
- falta de transações
- falta de isolamento adequado
- risco de concorrência na persistência
- estrutura de tabela inadequada para o fluxo operacional
- campos sem validação
- inconsistência temporal
- atualização fora de ordem
- falhas de auditoria de dados

6) UX OPERACIONAL CRÍTICA
Analise se a interface induz erro humano.
Quero que você trate como grave qualquer tela que:
- permita ambiguidade sobre qual pedido está em uso
- permita ações perigosas com 1 clique sem proteção
- não bloqueie requisições duplicadas
- não sinalize loading corretamente
- não trate atrasos/reconexão
- não deixe claro quando uma ação falhou
- permita usuário acreditar que concluiu algo que não foi salvo
- permita confusão entre item, quantidade, pedido ou operador
- não destaque estados críticos
- não trate corretamente falha de rede
- não proteja contra ações repetidas

7) OBSERVABILIDADE, AUDITORIA E PRODUÇÃO
Verifique:
- logs insuficientes
- logs excessivos
- ausência de trilha de auditoria
- ausência de correlation id
- dificuldade de rastrear incidente
- falhas sem contexto suficiente
- ausência de métricas
- ausência de alertas
- ausência de monitoramento dos fluxos críticos
- baixa capacidade de diagnosticar erro de produção

REGRAS IMPORTANTES
- Seja conservador em fluxos críticos
- Onde houver dúvida entre “mais flexível” e “mais seguro”, priorize o mais seguro
- Onde houver risco de erro operacional, endureça a regra
- Onde houver risco de concorrência, implemente proteção explícita
- Onde houver risco de atualização tardia, proteja contra stale state
- Onde houver risco de reprocessamento, torne a ação idempotente
- Onde houver risco de estado residual, limpe agressivamente o contexto
- Onde houver regra de negócio crítica, valide no backend
- Não confie no frontend para segurança
- Não aceite implicitamente dados do client
- Não mantenha lógica crítica espalhada
- Não deixe brechas para comportamento inconsistente
