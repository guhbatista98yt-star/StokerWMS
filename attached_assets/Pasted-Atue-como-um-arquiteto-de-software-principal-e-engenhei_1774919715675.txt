CONTEXTO FUNCIONAL CRÍTICO

Considere que este WMS opera com usuários reais, ritmo alto, pressão operacional e necessidade de precisão.
Nesses módulos, qualquer erro causa retrabalho, prejuízo, perda de confiança e quebra operacional.

Trate como HIPERCRÍTICO qualquer risco de:

1. PEDIDO RESSUSCITAR
Investigue profundamente qualquer cenário em que:
- pedido concluído reaparece
- conferência anterior reaparece em pedido novo
- item marcado em pedido antigo aparece em pedido atual
- estado de uma aba antiga volta para a interface
- navegação de volta ou cancelamento traz pedido já finalizado

2. LOCK INCORRETO
Investigue profundamente qualquer cenário em que:
- lockedBy não é limpo corretamente ao concluir ou cancelar
- pedido fica preso indevidamente
- pedido fica visível para quem não deveria
- mais de um operador acessa o mesmo pedido
- o frontend acredita que possui o lock quando o backend não garante isso
- o lock depende demais de estado local
- o lock não é protegido por backend e persistência adequados

3. STALE STATE / SESSÃO ANTIGA CONTAMINANDO A NOVA
Investigue profundamente qualquer cenário em que:
- promises tardias chegam depois da troca de pedido
- resposta antiga sobrescreve estado novo
- selectedWorkUnits antigo continua afetando a tela
- checkingTab, step, sessionVersion, token de sessão ou stores locais podem vazar para outro fluxo
- troca rápida entre pedidos causa contaminação
- navegação entre telas não limpa contexto corretamente
- localStorage / sessionStorage / stores transitórias mantêm dados críticos além do necessário

4. CONCLUSÃO AUTOMÁTICA INDEVIDA
Investigue qualquer lógica que:
- conclua conferência automaticamente ao atingir último item
- avance status sem ação explícita do operador
- permita finalização implícita
- finalize por efeito colateral de validação de item
- misture regra de conferência com regra de separação
- gere transição de status sem confirmação manual e rastreável

5. QUANTIDADE EXCEDENTE / ITEM EXCEDENTE
Investigue profundamente se existe qualquer inconsistência entre:
- bipagem
- digitação manual
- validação de item
- reset de item
- persistência de checkedQty
- reversão de excesso
- feedback visual para excesso
- preservação da quantidade válida já conferida

Verifique se o sistema pode:
- zerar quantidade válida por engano
- perder conferência válida após tentativa excedente
- tratar diferente a mesma regra quando vem de barcode versus input manual
- permitir excesso por corrida de requisições

6. MÚLTIPLAS ABAS / MÚLTIPLAS SESSÕES
Investigue profundamente:
- mesma conta em múltiplas abas
- mesmo operador em sessões paralelas
- operadores diferentes no mesmo pedido
- conflitos de atualização
- polling concorrente
- telas abertas em estados diferentes
- sessão antiga reaplicando mudança em pedido novo
- falta de invalidação/coerência entre abas

7. SEGREGAÇÃO POR EMPRESA / PERFIL / MÓDULO
Audite rigorosamente se existe qualquer possibilidade de:
- usuário da empresa A ver dados da empresa B
- operador acessar pedido que não deveria
- módulo ficar acessível sem permissão correta
- rota, pedido, conferência, balcão, endereço, romaneio ou nota ficarem cruzados entre empresas
- perfil antigo ou cacheado manter permissão indevida
- backend confiar apenas na UI para restringir acesso
